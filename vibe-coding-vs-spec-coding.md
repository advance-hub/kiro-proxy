# Vibe Coding vs Spec Coding

## 什么是 Vibe Coding？

Vibe Coding 由 Andrej Karpathy（OpenAI 联合创始人、前 Tesla AI 总监）在 2025 年初提出。

核心理念：开发者用自然语言向 AI 描述需求，AI 生成代码，开发者凭"感觉"（vibe）决定是否接受，不深入审查每一行代码。

### 特点

- 用自然语言驱动开发，几乎不手写代码
- 快速迭代，重结果轻过程
- 开发者角色从"编写者"变为"引导者"
- 依赖 AI 的代码生成能力，信任度高
- 出了问题倾向于让 AI 修，而不是自己 debug

### 适用场景

- 快速原型验证（MVP / PoC）
- 个人项目、side project
- Hackathon
- 探索性开发、技术调研
- 一次性脚本或工具

### 风险

- 代码质量不可控
- 可能引入安全漏洞
- 难以维护和扩展
- 团队协作困难
- 技术债务积累快

---

## 什么是 Spec Coding？

Spec Coding 强调在 AI 生成代码之前，先定义清晰、结构化的规格说明（Specification），让 AI 在明确的约束下工作。

核心理念：先想清楚，再让 AI 写。用规格文档作为 AI 的"合同"，确保输出符合预期。

### 特点

- 先写需求文档、接口定义、设计规格
- AI 按照 spec 生成代码，有明确的验收标准
- 开发者深度参与架构设计和代码审查
- 强调可测试性、可维护性
- 迭代过程有据可查

### 适用场景

- 生产环境项目
- 团队协作开发
- 长期维护的产品
- 对安全性、可靠性要求高的系统
- 需要代码审查和合规的项目

### 工作流程

1. 定义需求和约束
2. 编写接口规格 / API 设计
3. 制定验收标准和测试用例
4. 让 AI 按照 spec 生成实现
5. Review、测试、迭代

---

## 对比总结

| 维度 | Vibe Coding | Spec Coding |
|------|-------------|-------------|
| 开发速度 | 极快 | 较慢但可控 |
| 代码质量 | 不确定 | 较高 |
| 适合阶段 | 探索、原型 | 生产、交付 |
| 开发者角色 | 引导者 | 架构师 + 审查者 |
| AI 依赖度 | 高 | 中等 |
| 可维护性 | 低 | 高 |
| 团队协作 | 困难 | 友好 |
| 学习门槛 | 低 | 需要设计能力 |

---

## 实践建议

两者并不对立，可以结合使用：

- 项目初期用 Vibe Coding 快速探索方案、验证可行性
- 方向确定后切换到 Spec Coding 落地实现
- 个人项目可以偏 Vibe，团队项目偏 Spec
- 关键模块（支付、认证、数据处理）务必用 Spec Coding
- 无论哪种方式，都要有基本的测试覆盖
